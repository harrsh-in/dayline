# Dayline Product Guidance

Dayline is a simple, minimal productivity app built around tasks, habits, events, and calendar sync. Every decision made while working on this project should protect the core product promise:

> Help users see what they need to do, when they need to do it, without making planning feel like another task.

Dayline is not meant to become a heavy project management platform. It should feel calm, fast, focused, and easy to use from the first session.

---

## Product Identity

Dayline is a todo and personal planning app with three core pillars:

1. **Tasks** — things the user needs to complete.
2. **Habits** — routines the user wants to repeat consistently.
3. **Events** — scheduled commitments, both one-time and recurring.

The app also connects with Google Calendar and Microsoft Calendar so users can see their actual day in one place.

The product should compete directly with Todoist-style productivity apps, but with a sharper focus on simplicity, visual calm, and day-based planning.

---

## Product Promise

Dayline should make users feel:

- Clear about what needs attention today.
- In control of their tasks, routines, and events.
- Less scattered between todo lists and calendar apps.
- Confident that their schedule and tasks are not drifting apart.
- Comfortable using the app every day without needing setup rituals.

The product should avoid complexity that makes users feel like they are configuring a system instead of managing their day.

---

## Core Product Principles

### 1. Simple Before Powerful

Every feature should be understandable without explanation.

A user should be able to open Dayline and immediately understand:

- What is due today.
- What is scheduled.
- What habit is pending.
- What events are coming up.
- What needs action now.

Power features are allowed only when they do not damage the simplicity of the primary experience.

---

### 2. The Day Is the Center

Dayline should be built around the user’s day.

The default experience should answer:

> What does my day look like?

Tasks, habits, and events should not feel like separate worlds. They should work together as one daily timeline or planning surface.

---

### 3. Calendar Sync Should Feel Natural

Google Calendar and Microsoft Calendar integration should not feel like an advanced setting hidden away for technical users.

The user should be able to:

- Connect one or more calendars.
- See calendar events inside Dayline.
- Add Dayline tasks/events into connected calendars.
- Understand what is synced and what is not.
- Disconnect calendars safely.

Calendar sync must feel predictable. Users should never be surprised by duplicate events, missing events, or unclear sync behavior.

---

### 4. Minimal UX Is a Requirement, Not a Style Choice

Dayline must stay visually clean.

The interface should avoid:

- Dense dashboards.
- Too many panels.
- Overloaded sidebars.
- Excessive colors.
- Heavy borders.
- Complex settings screens.
- Feature-heavy empty states.
- Corporate productivity clutter.

The product should feel lightweight, direct, and polished.

---

## UX Direction

Dayline should feel closer to a calm daily planner than an enterprise task board.

The interface should prioritize:

- Quick task capture.
- Clear today view.
- Clean upcoming view.
- Simple habit tracking.
- Clear calendar event visibility.
- Low-friction recurring event creation.
- Fast rescheduling.
- Obvious completion states.
- Minimal visual noise.

Every main screen should have one primary job.

If a screen tries to do too many things, simplify it.

---

## Visual Direction

The design should use an appropriate color and font scheme that supports calm focus.

The visual system should feel:

- Minimal.
- Modern.
- Mature.
- Soft but not childish.
- Premium without being decorative.
- Clear in both light and dark modes if dark mode is added.

Avoid loud gradients, playful gimmicks, and overdesigned cards unless they improve clarity.

Recommended design feeling:

> Clean workspace, soft contrast, strong readability, quiet confidence.

## Feature Scope

### Core Features

Dayline should support:

- Task creation.
- Task completion.
- Task scheduling.
- One-time tasks.
- Recurring tasks.
- Habits.
- Habit completion tracking.
- One-time events.
- Recurring events.
- Calendar connection.
- Google Calendar integration.
- Microsoft Calendar integration.
- Multiple connected calendar accounts.
- Unified daily view.
- Upcoming view.
- Simple settings for connected calendars.

---

## Calendar Integration Behavior

Dayline should support connecting:

- One Google Calendar account.
- Multiple Google Calendar accounts.
- One Microsoft Calendar account.
- Multiple Microsoft Calendar accounts.

The user should be able to choose which calendars are visible inside Dayline.

Dayline should clearly separate:

- Dayline-created tasks.
- Dayline-created events.
- External calendar events imported from Google or Microsoft.

External events should be visible inside the app, but they should not automatically become Dayline tasks unless the user explicitly chooses that.

---

## Recurring and One-Time Items

Dayline should support both recurring and one-time items across the planning experience.

Examples:

- One-time task: “Submit invoice tomorrow.”
- Recurring task: “Review weekly goals every Monday.”
- Habit: “Drink water.”
- One-time event: “Client call at 4 PM.”
- Recurring event: “Team sync every Wednesday.”

Recurring behavior should be simple and predictable. Avoid complex recurrence builders unless truly needed.

---

## Competitive Positioning

Dayline competes with Todoist-style productivity apps.

The goal is not to copy feature-for-feature.

The goal is to compete on:

- Simpler daily planning.
- Cleaner UX.
- Better calendar visibility.
- Easier handling of tasks, habits, and events together.
- Less configuration.
- Less visual clutter.
- More confidence in what the user needs to do today.

Dayline should not become a clone. It should feel familiar enough to understand quickly, but distinct enough to justify using it.

---

## What Dayline Is

Dayline is:

- A personal productivity app.
- A todo app.
- A lightweight daily planner.
- A habit tracker.
- A simple event planner.
- A unified calendar companion.
- A calm place to organize the day.

---

## What Dayline Is Not

Dayline is not:

- A full project management suite.
- A team collaboration platform in the first version.
- A Jira replacement.
- A Notion replacement.
- A CRM.
- A heavy calendar app replacement.
- A complex productivity operating system.
- A dashboard full of charts and widgets.

If a feature makes the product feel heavier without making daily planning easier, it should not be added.

---

## Writing and Content Tone

All product copy should be:

- Clear.
- Human.
- Short.
- Calm.
- Direct.
- Useful.

Avoid:

- Corporate language.
- Motivational fluff.
- Overpromising.
- Technical terms.
- Empty productivity buzzwords.

Good copy examples:

- “Plan your day.”
- “Add a task.”
- “Connect calendar.”
- “No events today.”
- “This task repeats every Monday.”
- “Synced with Microsoft Calendar.”

Bad copy examples:

- “Optimize your productivity workflow.”
- “Empower your task orchestration.”
- “Unlock your peak performance.”
- “Configure your integrated planning ecosystem.”

---

## Decision Rule for New Features

Before adding any new feature, ask:

1. Does this help the user plan or complete their day?
2. Does this keep the app simple?
3. Can a first-time user understand it quickly?
4. Does this reduce scattered planning?
5. Does this belong inside Dayline, or is it just nice to have?

If the answer is unclear, do not add the feature yet.

---

## Product North Star

The product should always move toward this outcome:

> A user opens Dayline and immediately knows what their day looks like, what needs to be done, and what is already scheduled.

Everything else is secondary.
